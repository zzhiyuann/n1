# N1 Methods Whitepaper — v0 (draft)

> Status: draft. Purpose: (1) an auditable record of the methodology, and (2) the
> implementation specification for [`packages/N1Stats`](../packages/N1Stats).
> Open items: complete literature citations (N-of-1 trial reviews, AR(1) within-person
> models) and unify notation.

## 1. Problem definition

The fundamental question a personal-health tool should answer is not "what does my data
look like" but "**for me, does X cause Y?**" Population-level RCTs answer the average causal
effect; the individual-level question requires an **N-of-1 trial**: alternating intervention
and control within the same person, using the person as their own control.

N1 puts this methodology in your pocket: the user states a hypothesis → the system generates
a crossover design → pre-registration locks it → data is collected passively → a Bayesian
analysis runs → and a conclusion is reported with honest uncertainty.

## 2. Experiment design

### 2.1 Design space (v1)

- **Block crossover design.** An experiment consists of K pairs of blocks; within each pair,
  {intervention A, control B} is randomized (restricted randomization, ensuring balance and
  robustness to time trends). Default K = 2 (i.e. ABAB / BABA variants), each block 5–7 days.
- **Washout.** A w-day washout is inserted between adjacent blocks (set by the intervention's
  pharmacokinetic/physiological half-life; templates ship defaults, e.g. 1 day for caffeine).
  Washout-period data **does not enter the primary analysis**.
- **Honest statement of blinding.** Most lifestyle interventions cannot blind the subject. The
  protocol records "unblinded" explicitly and lists possible expectation effects as a limitation
  rather than pretending they don't exist.

### 2.2 Pre-registration

Before the experiment starts, the following are locked and hashed:

1. Hypothesis (directional, e.g. "caffeine after 14:00 lowers that night's HRV").
2. Primary endpoint (a single variable + aggregation, e.g. "mean nightly rMSSD").
3. Design parameters (K, block length, washout, start date).
4. Analysis plan (model, priors, decision rule — see §3).
5. Secondary endpoints (limited in number, explicitly marked exploratory).

Any change after the start is automatically downgraded to exploratory and presented separately
in the report. This is the core difference from typical quantified-self tools, and the source of
the conclusion's credibility.

## 3. Statistical model (N1Stats implementation spec)

### 3.1 Data structure

The primary endpoint is a daily measurement y_t (t = 1…T); each day carries a label
z_t ∈ {A, B, washout}. The primary analysis uses non-washout days only.

### 3.2 Model v1: hierarchical normal model of block means

For block j (j = 1…2K), with daily observations within the block:

```
y_tj = μ + δ·I(j is intervention) + b_j + ε_tj
b_j  ~ Normal(0, τ²)     # block random effect (absorbs slow drift: season, stress periods)
ε_tj ~ Normal(0, σ²)     # within-day noise
```

- **Quantity of interest:** δ (the within-person effect of the intervention on the primary endpoint).
- **Why not a t-test:** daily observations are correlated within a block (τ² > 0); treating days as
  independent samples produces spurious significance. The block random effect is a first-order defense
  against this autocorrelation.
- **v1 does not model AR(1):** with short blocks (5–7 days), AR(1) and the block effect are hard to
  identify jointly; v2 will evaluate replacing ε with AR(1) and compare calibration via simulation.
- **Carryover:** not modeled; handled at the design level by washout exclusion (classic N-of-1 practice).
  Interventions with insufficient washout are flagged at the template level.

### 3.3 Priors

- μ: Normal(user's historical baseline mean, historical variance) — estimated automatically from
  ≥ 14 days of retrospective HealthKit data before the start.
- δ: Normal(0, s²), with s = historical standard deviation (weakly informative, neutral on effect direction).
- σ, τ: Half-Normal(0, historical standard deviation) — v1 uses a grid/conjugate approximation (see §3.5).

### 3.4 Decision rule (locked at pre-registration)

Report three things; do not report p-values:

1. The posterior distribution of δ (presented primarily as a plot) with a 90% credible interval.
2. P(δ < 0) or P(δ > 0) (the posterior probability consistent with the hypothesized direction).
3. **ROPE decision:** a pre-registered region of practical equivalence (e.g. ±0.2× historical SD);
   report the posterior probability that δ falls inside the ROPE.

The text conclusion is generated by deterministic code (the LLM only polishes wording, never produces numbers):

- P(direction correct) ≥ 0.95 → "strong evidence".
- 0.80–0.95 → "preliminary evidence; consider extending the experiment".
- ROPE probability ≥ 0.90 → "evidence the effect is negligible".
- otherwise → "insufficient evidence; no conclusion".

### 3.5 Computation (v1)

The hierarchical normal model is fit with **Gibbs sampling** (all full conditionals are conjugate:
μ, δ normal; b_j normal; σ², τ² inverse-gamma): 4 chains × 2000 draws, with an R̂ < 1.01 convergence
check. Pure-Swift implementation, no external dependencies — a prerequisite for open auditability.

### 3.6 Validation (release gates)

- **Recovery test:** simulate data with known (δ, σ, τ) × 1000, checking that the 90% interval
  coverage ∈ [88%, 92%] and the posterior means are unbiased.
- **Null-effect calibration:** under δ = 0 simulations, the "strong evidence" false-positive rate ≤ 5%.
- **Degenerate cases with a known closed form** (τ → 0) are compared against the conjugate closed-form
  solution, with numerical error < 1e-6.

## 4. Data layer

- HealthKit public APIs only. Primary-endpoint candidates for v1: nightly HRV (Apple's SDNN-based
  measure), resting heart rate, sleep duration/efficiency.
- **Missing data:** the Bayesian model handles unbalanced blocks naturally; a block with > 30% missing
  data is flagged as low-quality and shown in the report.
- Local-first: all computation happens on-device; there is no network upload path.

## 5. Limitations (honest list)

1. No blinding → expectation effects cannot be ruled out; the report always includes this statement.
2. Multiplicity of the single primary endpoint is controlled, but exploratory endpoints still carry
   garden-of-forking-paths risk → presented separately.
3. Slow effects (e.g. exercise adaptation) are invisible under short blocks → templates restrict the
   applicable intervention types.
4. Device measurement error in HealthKit itself is not modeled (v2: a measurement-error layer).

## Appendix A · v0 implementation notes (findings from the first N1Stats recovery test)

1. **Variance priors.** The implementation uses InvGamma (conjugate, keeping Gibbs fully closed-form)
   rather than the Half-Normal sketched in §3.3; InvGamma(2, s²) has prior mean s² and is equivalently
   weakly informative.
2. **The τ² scale must be smaller than the σ² scale.** The first implementation used the same scale s₀²
   for both τ² and σ², and the recovery test immediately exposed systematic shrinkage of δ toward zero
   (bias ≈ +2.0 when true δ = −5). The cause is identifiability: a typical design has only J = 4 blocks,
   so the data cannot identify τ², and an overly large τ prior lets the block effects absorb the
   between-condition mean difference. Fix: set the τ² scale to (s₀/3)² (a weakly informative assumption
   that block-level drift ≤ 1/3 of day-level noise), and widen the δ prior SD to 1.5·s₀. After the fix,
   30 repetitions passed: bias < 0.8, 90%-interval coverage ≥ 24/30, and null-effect "strong evidence"
   false positives ≤ 4/30.
3. **Structural lesson.** With K = 2 block pairs, confounding between δ and (b̄_T − b̄_C) is intrinsic to
   the design; increasing the number of block pairs is the fundamental mitigation, and the app should
   honestly surface the uncertainty of K = 2 on the experiment-design screen.

## 6. Roadmap

- v1 (this document): block hierarchical normal + Gibbs + ROPE decision.
- v2: AR(1)-residual comparison, a measurement-error layer, binary/count endpoints (logistic/Poisson).
- v3: joint presentation of multiple endpoints, statistical review standards for a community template library.
